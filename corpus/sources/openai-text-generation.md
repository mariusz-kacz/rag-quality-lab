# Request shape and message roles

The OpenAI text generation guide describes how to use a large language model through the OpenAI API to generate text from a prompt. A model can generate prose, code, mathematical content, structured JSON, and other text responses. The guide recommends using the Responses API for direct text-generation requests.

## Basic Responses API request

A minimal text-generation request sends a model and an input prompt to the Responses API:

```python
from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    model="gpt-5.5",
    input="Write a one-sentence bedtime story about a unicorn.",
)

print(response.output_text)
```

The same request can be represented as JSON over HTTP:

```json
{
  "model": "gpt-5.5",
  "input": "Write a one-sentence bedtime story about a unicorn."
}
```

The response contains generated items in an `output` array. For a simple request, the array may contain one assistant message with `output_text` content. More complex responses can include tool calls, reasoning metadata, and other output items. The guide warns that it is not safe to assume text is always located at `output[0].content[0].text`.

Official SDKs may expose an `output_text` convenience property that aggregates text outputs into one string. This shortcut is useful for simple text-generation workflows, but robust applications should still understand that the canonical response shape is a list of output items.

For RAG systems, this means answer extraction should be explicit. If the application expects only answer text, use the SDK convenience property or parse the response output by type. If the workflow can include tool calls, citations, or structured output, do not assume a single fixed array position.

## Responses API over older chat workflows

The guide states that OpenAI has many models and APIs, and recommends the Responses API for text generation applications. It specifically notes that reasoning models perform better and show higher intelligence when used with the Responses API, and that applications using reasoning models should migrate to Responses.

For a RAG quality lab, this makes the Responses API the natural request surface for traced question answering. The request can include high-level instructions, user input, optional reasoning settings, selected context, and later tool or structured-output support without depending on older chat-specific assumptions.

## Message roles

The guide explains that developers can provide instructions with different levels of authority using the `instructions` parameter or an `input` array with role-tagged messages.

The main message roles are:

- `developer`: instructions provided by the application developer, prioritized ahead of user messages;
- `user`: instructions provided by the end user, prioritized behind developer messages;
- `assistant`: messages generated by the model.

The guide suggests thinking of developer and user messages like a function and its arguments:

- developer messages define system rules and business logic;
- user messages provide inputs and configuration to which the developer instructions are applied.

This distinction is important for RAG. The application should put durable behavior, citation rules, no-answer policy, context-use rules, and safety boundaries in developer-level instructions. The user's question and retrieved source content should not be treated as equal authority to the application rules.

## Instructions parameter and equivalent messages

The `instructions` parameter gives high-level instructions for the current generation request, including tone, goals, and examples of correct responses. Instructions supplied this way take priority over the prompt in the `input` parameter.

A compact request with instructions:

```python
from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    model="gpt-5.5",
    reasoning={"effort": "low"},
    instructions="Talk like a pirate.",
    input="Are semicolons optional in JavaScript?",
)

print(response.output_text)
```

The guide shows that this is roughly equivalent to using an input array with a `developer` message and a `user` message:

```python
from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    model="gpt-5.5",
    reasoning={"effort": "low"},
    input=[
        {
            "role": "developer",
            "content": "Talk like a pirate.",
        },
        {
            "role": "user",
            "content": "Are semicolons optional in JavaScript?",
        },
    ],
)

print(response.output_text)
```

For a context-grounded answer pipeline, the same pattern can separate durable system behavior from user input:

```python
response = client.responses.create(
    model="gpt-5.5",
    instructions=(
        "Answer only from the provided context. "
        "If the context is insufficient, say that the evidence is insufficient. "
        "Cite the source ids used."
    ),
    input=[
        {
            "role": "user",
            "content": "Question: What metric rewards relevant documents ranked early?\n\nContext:\n[source: trec-common-evaluation-measures]\nAverage precision rewards systems that retrieve relevant documents quickly.",
        }
    ],
)
```

The example keeps the application rule outside the user message. The user message contains the dynamic question and selected context.

# Instruction guidance and context-management implications

The guide defines prompt engineering as writing effective instructions so a model consistently generates content that meets requirements. It also warns that generated content is non-deterministic. Prompting is therefore a mix of art and science, and production applications should use tests and evaluation suites to monitor behavior as prompts or models change.

## Pin models and evaluate prompt behavior

For more complex applications, the guide recommends:

- pinning production applications to specific model snapshots to keep behavior consistent;
- building tests and evaluation suites that measure prompt behavior as prompts are changed or model versions are upgraded.

For RAG, this means the model version is part of the evaluation context. Retrieval quality can remain constant while answer behavior changes after a model upgrade. Golden-set evaluations should record the model, prompt version, context-building policy, and retrieval mode.

## Context and output handling

The guide's response-shape warning has direct context-management implications. Responses can include more than plain text: tool calls, reasoning-related data, and other items may appear in the `output` array. Applications should process the output by type and by contract, not by fragile positional assumptions.

RAG applications should decide which response contract they expect:

- plain answer text through `output_text`;
- structured JSON through structured outputs;
- answer plus citations parsed from a schema;
- tool-using workflows where tool calls are expected and handled separately;
- no-answer responses when evidence is insufficient.

Clear response contracts make traces easier to evaluate. They also reduce the chance that a tool call, reasoning artifact, or unexpected output item is mistaken for answer text.

## Instructions and conversation state

The guide notes that the `instructions` parameter applies only to the current response generation request. If an application manages conversation state with `previous_response_id`, instructions used on previous turns are not automatically present in the current context.

This matters for multi-turn RAG systems. Durable policies such as citation requirements, context-only answering, and no-answer behavior should be supplied on every request where they are required. A system should not assume that earlier instructions persist unless it explicitly includes or manages them.

For a traceable RAG workflow, each turn should record:

- the developer instructions used for that request;
- the user question;
- selected context chunks;
- retrieval mode and filters;
- model and reasoning settings;
- response output and citations;
- any excluded chunks or context budget decisions.

Without those fields, it is hard to reproduce or evaluate the answer.

## Prompt versioning in code

The guide recommends storing production prompts in application code rather than creating reusable prompt objects. Code-managed prompts support typed inputs, code review, tests, and the normal deployment process for changing model behavior.

For new text-generation work, the guide recommends:

- keep prompt builders in a small module near the feature they support;
- use typed function arguments or schemas for dynamic values such as customer data, files, or task options;
- pass generated `instructions` and `input` directly to the Responses API;
- add representative fixtures, tests, and evaluation checks before changing production prompts;
- roll out prompt changes through the deployment system, using feature flags or configuration for staged releases.

The source also notes that reusable prompt objects in the API are being deprecated. Prompt creation is scheduled to be de-emphasized beginning June 3, 2026, and `v1/prompts` is scheduled to shut down on November 30, 2026, according to the page's linked deprecation guidance.

For a RAG quality lab, code-managed prompts pair well with evaluation artifacts. Prompt changes can be reviewed like source changes, tested against golden questions, and correlated with trace outputs. This is safer than editing reusable prompt state outside the application's normal release and review path.

## Practical RAG request pattern

A compact RAG request should make the model's job explicit:

1. Put durable behavior in `instructions` or a developer message.
2. Put the user's question and selected context in the request input.
3. Keep retrieved context clearly labeled as source material, not instructions.
4. Require no-answer behavior when the context lacks evidence.
5. Require citations or structured fields when downstream evaluation needs them.
6. Record the model, prompt text, context, and response in a trace.

This pattern reflects the guide's core guidance: use the Responses API, understand the response output shape, separate higher-priority developer instructions from user input, and manage prompt behavior through code, tests, and evaluations.
