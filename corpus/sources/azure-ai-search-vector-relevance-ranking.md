# KNN, ANN, and similarity metrics

Azure AI Search ranks vector search results by looking for vectors similar to the query vector and scoring candidates by a configured similarity metric. The search behavior depends on how the vector field was indexed. A field can be searched exhaustively across the vector space, or it can be searched through an approximate nearest-neighbor index optimized for faster retrieval.

Only vector fields marked as `searchable` in the index, or specified through `searchFields` in the query, participate in vector search and scoring. This matters for RAG systems because the retriever can only rank fields that were intentionally exposed for search. Non-searchable metadata can still be useful for filtering, citations, or display, but it is not part of the vector nearest-neighbor comparison.

The source describes two algorithm families used in vector search:

- Exhaustive K-Nearest Neighbors, or KNN, performs a brute-force scan over the vector space and returns the exact `k` nearest neighbors for the query point.
- Hierarchical Navigable Small World, or HNSW, performs Approximate Nearest Neighbor search and is the ANN algorithm used by Azure AI Search.

Exhaustive KNN calculates distances between the query vector and candidate vectors without relying on a graph built for fast random access. Because it does not require the extra HNSW data structure, it does not consume vector index size quota in the same way. Its advantage is exactness: it returns the global nearest neighbors. Its cost is compute: scanning the full space is expensive, so exhaustive KNN is best for small or medium datasets, cases where precision is more important than query latency, or evaluation workflows where exact KNN provides a ground-truth set for measuring ANN recall.

HNSW is optimized for high-recall, low-latency search when datasets are larger or distributions are unknown or changing. During indexing, Azure AI Search builds additional graph structures that organize vectors into a navigable hierarchy. During query execution, the service navigates the graph to find likely nearest neighbors without scanning every vector. This design trades some exactness for throughput and latency and makes HNSW suitable for most large-vector-search scenarios.

HNSW has memory and quota implications. The graph depends on fast access to data points and therefore consumes vector index size quota. The source frames this as a balance: extra memory and indexing work support efficient query-time nearest-neighbor search.

Fields configured with HNSW can still be queried exhaustively by using the vector query request parameter `"exhaustive": true`. This is useful when an application needs exact results for a particular query or wants to compare approximate recall against exact neighbors. The reverse is not true: fields indexed as `exhaustiveKnn` cannot run HNSW queries because the HNSW graph structures were never created.

ANN is the broader class of algorithms that reduce the search space through specialized data structures or partitioning. ANN algorithms intentionally sacrifice some accuracy in exchange for scalable, faster retrieval. The tuning problem is therefore multidimensional: teams adjust algorithm parameters to balance recall, latency, memory use, and disk footprint.

Nearest-neighbor search works in an embedding space generated by a single embedding model or compatible model family. A query is embedded with the same model used to generate index embeddings, producing a vector in the same space. Similar concepts should cluster near each other, so retrieving relevant documents becomes the task of finding vectors closest to the query vector and returning the associated records.

For example, a hotel-related query should map near the cluster of vectors representing hotel documents. The similarity metric then determines which hotel vectors are most similar to the query and therefore which documents are most relevant. In a RAG pipeline, those matching documents or chunks become candidates for grounding a model response.

When a vector field is indexed for exhaustive KNN, the query is evaluated against all neighbors. When it is indexed for HNSW, the search engine traverses the HNSW graph and searches a subset of nodes within the vector index. This distinction is important for evaluation: exhaustive KNN can establish exact nearest neighbors, while HNSW must be evaluated as an approximate retrieval method whose recall depends on graph construction and query-time parameters.

## HNSW graph construction

During indexing, Azure AI Search constructs the HNSW graph so future queries can navigate efficiently. The source summarizes the process:

1. The service starts with an empty HNSW graph for a new index, or an existing graph when adding vectors to an existing index.
2. The graph has an entry point at the top level of the hierarchy. This is the starting point for indexing and later search.
3. Each vector is added as a node. Higher graph levels are more global, and lower graph levels are more granular.
4. Nodes connect to nearby neighbors. The `m` parameter controls the maximum number of neighbors for each node.
5. The `efConstruction` parameter controls how many candidate points are considered while forming graph connections.
6. The configured similarity `metric` determines distance between vectors when adding connections.
7. Some connections act as longer-range shortcuts across hierarchy levels, improving graph navigation.
8. Graph pruning and optimization can improve efficiency and navigability after vectors are indexed.

Higher `efConstruction` values cause the algorithm to consider more candidate connections during indexing. This can create denser local neighborhoods in the graph and improve recall, but it can also increase indexing cost and memory pressure. The default value called out by the source is 400, with a range from 100 to 1,000.

## HNSW query navigation

At query time, HNSW navigates the hierarchy to find nearest neighbors:

1. Search starts at the top level of the hierarchical graph.
2. The algorithm traverses level by level toward lower, more granular levels.
3. Candidate nodes closer to the query vector are selected using the configured distance metric.
4. The search space is pruned by maintaining likely nearest-neighbor candidates rather than scanning every node.
5. The `efSearch` parameter controls the length of the candidate queue considered during traversal.
6. As the algorithm reaches lower levels, it refines the candidate set by considering more local neighbors.
7. Search completes when the requested number of nearest neighbors has been identified or another stopping condition is reached.

The query-time `k` parameter governs how many nearest neighbors the query asks the system to return. In a RAG system, `k` is not just a search parameter. It determines how many candidate chunks are available for downstream context assembly, reranking, citation selection, or no-answer decisions.

## Similarity metrics

After candidate vectors are found, a similarity metric compares each candidate vector with the query vector. The ordered set of strongest matches becomes the ranked vector result set.

Azure AI Search supports three metrics in the source article:

- `cosine`: measures the angle between two vectors and is not affected by differing vector lengths. Azure OpenAI embedding models use cosine similarity, so Azure OpenAI embeddings should use `cosine` in the vector configuration.
- `dotProduct`: measures both vector length and angle. For normalized vectors, dot product is identical to cosine similarity, and can be slightly more performant.
- `euclidean`: also known as `l2 norm`, measures the length of the vector difference between two vectors.

Metric choice must match the embedding model and data preparation. If embeddings are expected to be compared by cosine, using a different metric can change ranking behavior and degrade retrieval quality. If vectors are normalized and the system can use dot product efficiently, dot product may be an equivalent scoring choice. Euclidean distance can be useful when distance in vector space is the intended model of similarity.

When a request runs multiple vector queries in parallel, or combines vector and text queries in a hybrid request, Azure AI Search uses Reciprocal Rank Fusion to compute the final combined result ranking. This means the final ordering in a hybrid or multi-vector query is not simply the raw vector score from one vector comparison. For RAG evaluation, this distinction matters because a result's final rank might reflect fusion across several ranked lists.

# Score interpretation, k selection, chunk tuning, and hybrid ranking

Azure AI Search assigns an `@search.score` to each returned match. The highest-scoring matches are returned as the `k` vector results requested by the query. For vector search using the `cosine` metric, the documented `@search.score` range is from 0.333 to 1.00.

The `@search.score` for cosine is not the raw cosine similarity between the query vector and document vector. Azure AI Search transforms the value so that it is usable as a monotonically decreasing ranking score: as similarity gets worse, score values decrease. The documented transformation defines score as:

```text
@search.score = 1 / (1 + cosine_distance)
```

The source also distinguishes cosine similarity from cosine distance:

- cosine similarity is the cosine of the angle between two vectors;
- cosine distance is `1 - cosine_similarity`.

If an application needs the original cosine similarity value, for example to set a custom threshold for trimming weak results, it can convert from the search score back toward similarity. Compact form:

```text
cosine_distance = (1 - score) / score
cosine_similarity = 1 - cosine_distance
```

This score interpretation is important for RAG quality work. A score threshold should be based on the meaning of the transformed search score, not on the assumption that `@search.score` is directly equal to raw cosine similarity. It is also important to compare scores only within compatible retrieval modes, metrics, embedding models, and query configurations.

## Selecting k for RAG

The query-time `k` value controls how many nearest-neighbor results the vector query returns. A small `k` can keep downstream prompts compact and reduce distracting context, but it can miss needed evidence. A larger `k` can improve recall and give a chat model more grounding candidates, but it increases context assembly work and can add irrelevant or redundant text.

The source explicitly recommends increasing `k` when the application sends search results to a chat model and the current result set is not sufficiently relevant. In RAG systems, `k` should be tuned together with:

- the context token budget;
- chunk size and overlap;
- whether a reranker or semantic ranker is used after vector retrieval;
- whether the query is simple lookup, comparison, multi-hop, or open-ended;
- the desired no-answer behavior when evidence is weak.

`k` should not be treated as "the final number of chunks in the prompt" unless the system has no downstream filtering. A more robust RAG pipeline may retrieve a larger candidate set, rerank or filter it, enforce source diversity, remove duplicate chunks, apply security trimming, and then choose the final context.

## Chunk size and overlap tuning

The source's first relevance-tuning recommendation is to try different chunk size and overlap settings. It specifically recommends increasing chunk size and ensuring enough overlap to preserve context or continuity between chunks when results are not relevant enough.

Chunking affects vector relevance because embeddings are computed over the chunk text. If a chunk is too small, it may omit terms or surrounding context needed for the vector to represent the answer. If a chunk is too large, it may mix unrelated topics, making the embedding less specific and the retrieved text harder to fit into a prompt. Overlap can preserve continuity at boundaries, but too much overlap creates redundant candidates and can waste prompt space.

For a RAG quality lab, useful chunk-tuning questions include:

- Does the chunk contain enough surrounding context to answer a likely question?
- Does the chunk mix multiple unrelated topics that should be embedded separately?
- Are titles, headings, or source identifiers included so retrieved chunks are understandable alone?
- Does overlap prevent boundary cuts without causing many near-duplicate hits?
- Can the top `k` retrieved chunks fit within the prompt budget after citation metadata is included?

The source does not prescribe one universal chunk size. Instead, it treats chunking as a relevance lever to test when retrieved results are poor.

## HNSW parameter tuning

For HNSW fields, the source recommends trying different `efConstruction` levels to change the internal composition of the proximity graph. Higher values consider more candidate neighbors during indexing and can improve graph quality and recall, at the cost of more work and resource use during indexing.

The source also describes `efSearch` as the query-time candidate queue length used while navigating the graph. Higher `efSearch` values generally allow broader exploration and can improve recall, but they can increase latency. The practical tuning workflow is to compare latency and recall at representative settings rather than assuming one configuration is optimal for every corpus.

Exhaustive KNN remains useful as an evaluation baseline. Because exhaustive KNN returns exact nearest neighbors, it can be used to create a ground-truth nearest-neighbor set and measure whether the HNSW configuration is retrieving the same or sufficiently similar results.

## Hybrid ranking and semantic ranking

Vector queries do not have all of the same relevance-tuning features as classic text search. The source notes that vector queries do not use specific tuning features such as scoring profiles or field and term boosting. If pure vector results are not good enough, the recommended options are to adjust query configuration, change chunking, tune HNSW parameters, increase `k`, or try hybrid queries with semantic ranking.

Hybrid search combines text and vector retrieval signals. In Azure AI Search, when vector and text queries are combined, Reciprocal Rank Fusion merges ranked lists into a final ranking. The source states that hybrid queries with semantic ranking consistently produced the most relevant results in benchmark testing.

For RAG systems, hybrid ranking is useful because vector search and lexical search fail in different ways. Vector search can match concepts that do not share exact words. Lexical search can preserve exact identifiers, names, codes, error messages, and rare terms. Semantic ranking can then improve the ordering of candidates before they are passed to a model.

Hybrid ranking also changes evaluation. A result returned by hybrid search can be present because it ranked well lexically, semantically, vectorially, or through fusion. Evaluation should therefore record retrieval mode, query configuration, `k`, semantic ranking use, and any score fields needed to interpret why the final context was selected.

## Practical relevance workflow

The source's tuning guidance can be turned into a practical loop:

1. Confirm that the correct vector fields are marked `searchable` and used by the query.
2. Confirm that query embeddings and index embeddings are produced by the same model or compatible embedding configuration.
3. Choose the similarity metric that matches the embedding model, such as `cosine` for Azure OpenAI embeddings.
4. Start with HNSW for larger datasets and use exhaustive KNN for small datasets or evaluation baselines.
5. Inspect top results and scores for representative questions.
6. Tune chunk size and overlap when retrieved chunks are too narrow, too broad, or missing boundary context.
7. Increase `k` when a chat model needs more candidate evidence.
8. Tune HNSW parameters such as `efConstruction` and `efSearch` to balance recall, latency, and resource use.
9. Use exhaustive KNN to check approximate recall when retrieval quality is uncertain.
10. Test hybrid search with semantic ranking when pure vector ranking is insufficient.

For this project's corpus, the article is most useful as a vector retrieval relevance reference. It explains the difference between exact and approximate nearest-neighbor retrieval, the role of HNSW graph parameters, the meaning of similarity metrics and transformed scores, and the practical levers that influence whether retrieved chunks are good enough to ground a RAG answer.
